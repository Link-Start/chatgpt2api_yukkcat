# 控制台问题审计与下一步决策

更新日期：2026-06-05

这份文档用于收口当前 Vue 控制台重写的问题清单。它不是流水账状态，而是把用户反馈逐条拆成：当前事实、完成程度、后续决策和实施顺序。后续改代码前先对照这份文档，避免只修一个点，漏掉同类问题。

## 总体结论

当前 `web-vue/` 已经能作为第一版控制台骨架运行，账号、日志、图片管理、设置、代理和概览中心都接了真实接口或可用 adapter。但这还不是“功能完全对齐原版”的状态。

已经基本完成的是：页面骨架、账号分页、账号导入/导出入口、批量进度弹窗、日志主表、图片管理分页、设置页保存、代理 profile 的单代理 URL 管理。

已经补出第一版结构但还没完全验收的是：账号组、多节点代理组、账号组绑定代理组、代理组 round-robin 选点和代理组测试写回。

还没有完成的是：真实请求失败自动写回代理节点冷却、微软/webfree 注册机接入、支持模型真实场景 smoke、统一 UI 组件抽象、真实代理和真实账号链路 smoke。

最关键的架构缺口是两个：

- 代理组第一版接口已经有了，但真实上游请求失败还没有统一写回节点健康状态。
- 账号组第一版接口和管理 UI 已经有了，但来源自动分组、来源级刷新和真实批量绑定 smoke 还没完成。

## 概览中心

### 已做

- 已保留 6 张原版语义图表：模型请求分布、调用趋势、成功率趋势、平均响应时间、模型调用占比、模型使用排行。
- “最近失败日志”面板已移除，失败排查统一进入日志管理。
- 账号卡片已改成：账号总数、正常账号、限流账号、异常账号、禁用账号、剩余额度。
- 卡片模板现在不渲染解释文字，避免出现“账号池中的总数量”这类说明。
- Header 接口信息已经有聊天模型和图片模型展示入口。
- 剩余额度主值现在只显示有限额度总和、`不限` 或 `0`；无限额度账号数量放在二级小字，不再拼成 `数字+数字∞`。
- Header 接口信息打开时会优先读取 `/v1/models`，失败时再回落到 settings/catalog。

### 半做

- 支持模型口径还没有最终对齐。
  - `D:\gemini2api\frontend` 的模型逻辑是先读配置/catalog；当前 Vue 控制台已补 `/v1/models` 动态读取。
  - 还没有用真实账号池验证 Header、文档页、图片模型选择页在不同账号类型下是否完全一致。
  - 后端 `/v1/models` 会根据账号池动态追加 `gpt-image-2`、`codex-gpt-image-2` 和 `plus/team/pro-codex-gpt-image-2`。

### 未做

- 没有完成“支持的模型完全跟原版交互一致”的验收。
- 没有做空数据场景和真实模型列表场景 smoke。
- 侧边栏选中态已经改过一轮，但用户仍觉得不协调，需要按整套视觉重新调，不算完成。

### 决策

- 图表维度继续按模型，不做“图片请求分布”这种单独口径。
- 剩余额度主值只显示一个清晰数字或“不限”，无限账号数量放副信息或 tooltip，不再拼成 `数字+数字∞`。
- 支持模型以接口返回为准，前端 fallback 只作为后端缺字段时的兜底。

## 账号管理

### 已做

- 列表已支持分页，默认 20 条，并通过 localStorage 记住页大小。
- 顶部入口已收成“导入 / 添加”“批量操作”“导出”三类。
- 导入弹窗已覆盖 OAuth、Access Token、Session JSON、Codex JSON、CPA JSON 文件、远程 CPA、Sub2API。
- 批量刷新、恢复、重置、启用、禁用、删除都走同一个进度弹窗。
- 批量停止现在只能停止后续批次，不能中断后端已经执行中的当前批次。
- Access Token 列表里脱敏展示，点击复制完整 token；最新决策是不在悬浮中显示完整 token。
- 异常、限流、禁用行的颜色按状态分类显示，已修过“禁用再启用后异常底色丢失”的一类问题。
- 编辑账号时 proxy 前台目标支持：全局代理、强制直连、代理组、自定义代理。

### 半做

- “恢复异常账号”功能已接 `/api/accounts/re-login`，但含义没有在产品上讲清楚。
  - 它不是微软/webfree 注册机。
  - 它是尝试用已有 OAuth refresh token 重新登录或恢复账号。
  - 没有 refresh token 的纯 Access Token 账号不一定能恢复。
- “刷新账号”和“恢复账号”语义还需要在 UI 上拆清楚。
  - 刷新账号：用当前 token 查账号信息、额度、状态。
  - 恢复异常账号：尝试 OAuth 重登或 refresh token 换新。
  - 微软/webfree：属于账号注册/导入模块，不属于刷新或恢复。
- 导入、导出、批量菜单已经收拢到同一个浮层菜单组件，顶部操作菜单最小浮层宽度、分割线和互斥打开逻辑已统一；账号页顶部工具栏已拆成账号组绑定、账号操作、低优先级刷新三类，刷新列表不再混在导入/批量/导出主操作组里。
- 编辑账号弹窗里的代理区已做第一轮减负：代理模式只负责选择，下面只显示一个“当前代理”摘要；刷新代理配置只在 profile/group 模式展示，强制直连不再显示测试代理按钮。
- 批量动作目前是前端 20 个一批调后端，不是后端统一批处理任务队列。可用，但大号池下不算最佳。

### 未做

- 账号组基础模型和管理弹窗已接入，但还没有做真实批量绑定 smoke。
- 账号组可以绑定默认代理组，但真实图像请求链路里的组级代理优先级还需要继续 smoke。
- 没有按来源自动分组，例如微软组、域名邮箱组、Codex 组、手动 token 组。
- 没有微软/webfree 注册机接入。
- 没有“单独刷新微软账号”这类来源专属刷新动作。
- 没有真实账号 smoke：OAuth、CPA、Sub2API、批量刷新、恢复异常、导出下载、代理测试都还没在真实对象上跑完。

### 决策

- 添加账号和导入账号放在同一个入口里是对的。
- 微软/webfree 不要硬塞进普通导入，应作为“账号来源模块”接入，后面可以在导入菜单中出现“从微软注册机导入/刷新”。
- 账号组不能只靠前端筛选伪装；当前已接入第一版真实模型，后续继续补来源自动归组、组级刷新和真实链路 smoke。
- 大号池下批量动作建议后端任务化，前端只启动任务和轮询进度。

## 系统设置

### 已做

- 设置保存、刷新、基础连接、全局代理、图片保留天数、账号策略、图片存储、备份、CPA、Sub2API 接口都已接入。
- `basic.proxy` 兼容写回说明已从 UI 移除。
- 2026-06-04 曾尝试重排设置页和统一 switch 行，但用户反馈“不如之前”，该方向已回退；当前设置页先保持原来的紧凑表单布局，不再把试验稿当成完成项。
- 设置页顶部临时统计和 `basic.proxy` 长说明已移除，避免把内部兼容字段暴露给用户。
- 2026-06-05 设置页回到原版“配置面板”心智：一个大圆角面板包住设置分组，刷新/保存放在面板头部，内部继续按基础连接、图片链路、账号并发、缓存、提示词、图片存储和备份分组。

### 半做

- 当前设置页已恢复到用户更能接受的包裹式旧布局口径，但还没有抽成统一设置组件。
- 图片轮询、图片账号并发、缓存、备份等高级项是否继续外露还没有最终定稿，后续要按“常用项优先、内部参数少解释”的规则重排。
- switch、checkbox、输入行还没有和账号/日志/图片管理的所有操作控件形成统一规范。
- 全局代理、账号组代理组、账号单独代理的优先级已有后端第一版支持，但设置页只应该展示全局代理，不应把兼容字段或长说明摊给用户。

### 未做

- 没有把所有 switch、输入行、说明文案统一成一套设置组件规范。
- 没有最终决定监控设置是否保留在第一版主导航。
- 没有完成设置页多尺寸浏览器视觉 smoke。

### 决策

- 设置页不再用“标题卡 + 外部三列散块”的结构；主设置必须包在一个配置面板里，内部再分组。
- 设置页主视图原则上只保留常用配置：上游 Base URL、全局代理、图片保留天数、账号策略、图片存储、提示词/敏感词。
- 图片轮询、缓存、并发和备份细节后续仍可继续按高级项减负；外部账号源当前已经收成列表 + 新增/编辑弹窗，默认不常驻显示密钥输入框。
- 代理优先级固定为：账号单独代理或直连 > 账号组绑定代理组 > 全局代理 > 直连。
- `basic.proxy` 只作为后端兼容字段，不在界面上解释。
- 设置页不显示临时统计，不解释内部读取路径，不把“监控设置”放进第一版主配置。

## 代理管理

### 已做

- 全局代理保存和测试已接。
- 多节点代理组 CRUD、节点列表、round-robin 选点和节点测试写回已接。
- 当前前台方案以 `group:<id>` 代理组引用为准。
- 账号组可以绑定默认代理组，后端解析优先级已接入。
- 编辑弹窗保存后关闭、确认弹窗层级压住编辑弹窗的问题已修过一轮。
- 代理页第一轮信息层级已收口：顶部不再放测试目标这类看板卡片；用户可见结构改成全局代理、代理组两块。
- 2026-06-05 已移除用户侧旧单代理区，前台不再把旧 profile 作为代理管理方案展示。

### 半做

- 真实代理连通性还没 smoke。
- 真实图像请求链路使用账号 `group:<id>` 或账号组默认代理组的端到端行为还没 smoke。
- 节点健康写回目前主要来自代理组测试接口；真实上游请求失败时的节点失败摘除/冷却写回还需要继续补。
- 代理测试失败时只能显示后端 `curl_cffi` 错误，例如 `Recv failure: Connection was reset`，还没有进一步诊断是代理认证、协议、目标站、出口 IP、DNS 还是供应商限流。

### 未做

- 没有按账号 stickiness 选择固定节点，避免同一账号频繁换出口。
- 没有把真实请求失败统一写回到代理节点健康状态。
- 没有代理故障诊断分层，例如认证失败、网络重置、DNS、目标站拒绝、供应商限流。

### 决策

后续代理模型应升级为：

- `proxy_groups`：代理组，字段包括 id、name、strategy、enabled，已接第一版。
- `proxy_nodes`：代理节点，字段包括 id、url、enabled、last_latency_ms、fail_count、last_error、cooldown_until，已接第一版。
- 选择策略第一版用 round-robin，先跳过 disabled/cooling 节点，已接第一版。
- 失败时记录错误并进入冷却，冷却后自动恢复候选：测试接口已记录失败，真实请求链路写回仍待补。
- 代理解析优先级为：账号单独代理或 direct > 账号组绑定代理组 > 全局代理，已接第一版。
- 前台只维护代理组；旧 profile 兼容能力留在后端，不作为控制台主流程。

### 半做

### 未做

- 没有抽成统一 `nanocat-ui` 或本项目通用卡片组件。
- 没有自定义日期选择组件。
- 没有完成真实大日志量下的性能 smoke。
- 没有决定是否把状态、类型、模型、账号进一步做成真正多选筛选。

### 决策

- 日志列表主视图保持旧版简洁，不把 request/error 塞回表格。
- 失败定位靠详情抽屉。
- 常用筛选保留：只看失败、图生图、文生图、文本回复无图、关键词、日期；快捷项放下拉，关键词和日期常驻。
- 其他筛选放高级筛选。
- 运行日志不解释来源细节，后续部署侧再配置 stdout/stderr 文件。

## UI 组件统一

### 当前事实

- 现在页面之间有三类不同卡片和面板风格：概览图表卡、账号/日志表格卡、图片管理工具卡。
- 用户觉得几种卡片都各有不错的地方，但整体不统一。
- 账号页顶部“账号组绑定”“导入 / 添加”“批量操作”“导出”、底部批量栏和行内“更多”已经统一到本项目 `FloatingActionMenu`；顶部四个浮层入口只传统一 `trigger-class`，宽度由菜单内容自适应并至少不小于触发按钮，互斥打开、无勾选账号组和分割线样式由组件内部负责。
- 下拉方向规则已经收口到 `FloatingActionMenu`：默认往下打开，只有窗口下方空间不足且上方空间更充足时才往上翻；长菜单使用内部滚动，避免“选择账号组”“导出”和行内“更多”在正常位置默认向上弹。
- 账号组不再只停留在筛选项：账号页已增加“账号组管理”弹窗，负责创建、编辑、启用/停用、删除账号组，并可绑定默认代理组；筛选下拉仍只负责筛选，批量绑定入口只负责把选中账号写入已有账号组。

## 数据存储边界

### 当前事实

- 本地账号和 auth key 已切到 SQLite。
- 配置仍主要在 JSON。
- 调用日志、运行日志仍是日志服务和文件/内存来源。
- 图片索引和标签仍按现有图片服务/文件元数据路径处理。

### 决策建议

- 1 万账号：本地单实例 SQLite 可以继续用，前提是账号列表服务端分页/索引要跟上。
- 多容器生产：不要三个容器各用一套 SQLite 管同一业务，应考虑 PostgreSQL 或明确每个容器独立号池。
- 日志和图片元数据是否进数据库，先看真实卡顿点，不要一次性全迁。
